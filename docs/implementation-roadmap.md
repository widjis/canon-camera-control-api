    - `POST /v1/sessions` returned `201` with an active lease
    - `POST /v1/captures` with `captureTarget: memoryCard`, `downloadToEdge: true`, `keepOnCamera: true` returned `202` with `jobId a542b795-7148-4c80-b81f-319862c6500d`
    - `GET /v1/jobs/{jobId}` reached `status: succeeded` with a clean `cameraPath` (`/store_00020001/DCIM/100CANON/IMG_0256.JPG`, no parsing-bug suffix) and a saved `asset` (`data/media/r50-card-20260714-090420.jpg`, 5.9MB, kept on camera)
  - post-completion addition (2026-07-14): added `GET /v1/media/{assetId}/content` in `src/app.ts` to stream a stored media asset's raw bytes (the existing `GET /v1/media/{assetId}` only ever returned metadata, with no way for a client to fetch the actual file) — needed so external clients (e.g. a browser front end) can retrieve captured images; `docs/openapi.yaml` updated with the new path; `npm run build` and `npm test` passed; verified live against a real captured asset (`81c8a6fc-c23e-4fa2-bdc6-baebb639f2c1`) — response was a valid 5.9MB JPEG with matching `Content-Length` and correct EXIF (Canon EOS R50, 6000x4000)
  - post-completion fix (2026-07-14): `GET /v1/camera/preview` was returning `500 PREVIEW_NOT_FOUND` on every call on this Windows development host — `gphoto2 --capture-preview`'s internal `thumb_` filename derivation only splits on `/`, so the backslash paths produced by Node's `path.join` on `win32` were never recognized as having a directory component; fixed in `capturePreview()` (`src/gphoto2.ts`) by normalizing the generated filename to forward slashes before passing it to gphoto2 (a no-op on Linux/macOS, so no behavior change on the production edge target); `npm run build` and `npm test` passed; verified live against the Canon EOS R50 on this device — 5 consecutive calls all returned `200` with a valid ~200KB 960x640 JPEG, averaging ~1.0-1.2s per call (see `docs/integration-contracts.md` for the preview-latency implication: there is no continuous video preview, only single-frame polling); no `docs/openapi.yaml` change required since the endpoint's contract was unchanged, only its Windows-specific failure was fixed
